## Session 1: Project Initiation

**Date:** 2026-03-02

### Questions & Answers

**1. What is the goal of the project?**
Two layers:

- **Deliverable:** A personal all-in-one app for budgeting, journaling, goals, tasks, decisions, and life management.
- **Underlying motivation:** Extreme curiosity leads to information overload and anxiety about losing track. LLMs now make it possible to not just collect personal data, but meaningfully process it. The app serves two goals: (1) accelerate personal growth by having an intelligent feedback loop on your own life, and (2) create an objective record of that growth to replace vague feelings with verifiable history. A third key driver is **data sovereignty** — every need this app addresses is served by existing products that monetize user data; building it yourself keeps that data under personal control and positions well for the coming age of AI.

**2. Who will be the primary user/consumer (who benefits)?**
Solely the user themselves. No multi-user or sharing features needed.

**3. What does success look like?**
Daily use as a first choice, not a fallback. The test is: would you reach for it naturally? Concrete example — tax season: if finances had been tracked for the past year, you could file with confidence knowing exact spend, reliable data, archived invoices/receipts/images. Two dimensions of success:
- **Usage:** It's built, used, and *wanted* — the preferred tool for everything it's intended to cover
- **Adaptability:** It can grow (and shrink) with evolving needs over time — not a static product
Core requirement underlying both: the data being collected is actually being processed in a way that serves the original purpose (growth acceleration + verifiable record).

**4. What are the time/urgency expectations?**
- **MVP:** End of March 2026 (~4 weeks)
- **Full-featured prototype:** Before September 2026 (~7 months)
- **Final product:** Doesn't exist — this is an evergreen personal tool that will always be iterated on; no planned end state.

**5. Why does this project matter to you right now? What's driving this need?**
The idea has existed for a while but now the pain is acute and the excuses are gone:
- **Active pain:** Tax season is forcing a painful document hunt that better organization would have prevented
- **Ongoing loss:** Consuming interesting books and ideas daily but losing them to distraction before implementing anything
- **Enablers removed:** A website is already set up to track development, and LLM coding agents make building this solo realistic in a way it wasn't before
- The honest answer: should have started a year ago. Starting now because there's no good reason not to anymore.

## Phase 2 Known Limitations (post-implementation 2026-05-17)

Items found via the end-of-Phase-2 audit that are *known* and either intentionally deferred or scoped to a later phase. Documented here so code review + Phase 3 work has them on the radar.

- **JournalFile projection covers only 3 of 16 budget event types** (`TransactionRecorded` + `AccountAdded` + `ExchangeRateRecorded`). The remaining 13 (Categorized, Tagged, Updated, Deleted, Cleared, Merged, Budget*, Recurring*, ...) don't update the on-disk hledger file. As soon as you delete/merge/tag a transaction, the file diverges from SurrealDB state. No regenerate-on-mutation path yet. **Fix scope:** add per-event handlers + a `regenerate_from_events` fallback path for events that can't be appended (mutations). Phase 4/5 polish.
- **The private bank driver's `description` field is a fragile fallback chain.** Different upstream transaction types yield inconsistently-shaped descriptions: card payments show merchant name, transfers show recipient, position adjustments show just a code. Users will see this inconsistency during review.
- **No frontend bindings for the 12 budget Tauri commands.** `record_transaction`, `list_transactions`, `add_account`, etc. are wired into `invoke_handler` but no Dioxus page calls them. Currently orphan code only callable from DevTools. **Fix:** Phase 3 capture-flow UI work.
- **`tracing-subscriber` env filter not unified across Tauri + server.** Two binaries, two `.init()` calls, two default filters. Cycle 4 polish should consolidate.
- **`imap-proto` future-Rust warning** — transitive dep of `imap` crate uses syntax some future Rust will reject. Waiting for upstream maintainer release.
- **The private bank source's first-run still needs interactive OTP entry.** After a password rotation, the saved session was invalidated; until interactive re-auth provides a fresh OTP, the source fails every tick (driver logs `OTP required`, scheduler backs off exponentially capped at 1h). **Fixed since (3.5a):** in-app "session expired, click to re-auth" affordance.
- **No surface for "session expired" alerts** — failure was only visible via server logs. Pairs with the previous item (both addressed by 3.5a).
- **SurrealDB tempfile race in tests** — known + documented in `~/.claude/projects/.../memory/project_known_bugs.md` (reproduced 3× during this session). Tests pass on rerun.
- **`cargo tauri dev` runtime not smoke-tested this session.** Only `cargo run -p omni-me-server` got the boot smoke test. Tauri startup path changed (auto-import wiring removed) but app wasn't actually booted. **Fix:** verify on first Phase 3 UI work.
- **The API source's `fxRate` and `totalFees` surface as posting tags, not typed fields** — see Session 5 log entry. Phase 4 polish could promote to typed `FxRate` + separate fee posting if a reliable direction convention is confirmed.
- **Auto-import source health bugs (surfaced 2026-05-18 device test)** — three independent issues observed in `/auto_import/status` on a running server:
  - **`gmail_work` + `yahoo`: intermittent `save cursor: Connection uninitialised`.** Same error in both; only fires on ticks that actually have an email to save cursor state for (most ticks have 0 events and succeed). Likely a SurrealDB connection-state issue specific to the cursor-store write path. The handlers still consume the events; it's the cursor save that fails.
  - **`gmail_personal`: manual tick returns HTTP 502 after successfully producing a batch.** The `AutoImportBatchProposed` event is appended (verified — phone synced and rendered the walmart receipt batch correctly during device test) but the HTTP handler returns Bad Gateway. Some downstream error after the append.
  - **The private bank driver: OTP required (genuine credential expiry).** Saved session invalidated; needs interactive re-auth. Already known per the earlier session-expired entry above — re-tripped after the next rotation.

## Phase 3 Known Gaps (post-implementation 2026-05-17)

Capture-flow bundle (3.1 + 3.2 + 3.4 + 3.5 + 3.6) ships at the UI surface today; 3.3 is partial (manifest entry only, gitignored). End-to-end **mock** flow visually verified via Playwright; **real-server + Android paths NOT smoke-tested**. This section is the honest gap list — what's intentionally deferred vs. what would burn the user.

### Likely to bite soon

- ~~**Attachment bytes are dropped after extraction.**~~ **RESOLVED 2026-05-17 via Phase 3.7.** Server `/documents/extract` now accepts `?attach=true`, stores bytes content-addressably, returns `AttachmentRef`; Tauri client mirrors to `<app_data>/attachments/<sha256>` LRU cache and threads the ref through to the `TransactionRecorded` event. `TransactionForm` shows an "Attachment saved" line in lieu of the deferred thumbnail (thumbnail render lives in Phase 4 transaction detail view, which `fetch_attachment` already supports).
- **No balance validation in `TransactionForm`.** The form doesn't check that postings sum to zero per commodity. User can submit `Expenses:Foo 10 CAD` + `Assets:Bar 10 CAD` (both positive); backend will store an unbalanced transaction. [[project-unmatched-account-pattern]] depends on `Unmatched` clearing to zero — an unbalanced submission breaks that invariant. **Fix:** validate per-commodity sum before `invoke_record_transaction`; auto-balance by injecting an `Unmatched` posting if the user wants one but didn't enter it.
- **Amount parser laxness.** Frontend uses `f64::parse` for amount validation — accepts `1e2`, `0.1.2`, scientific notation. Backend's `rust_decimal::Decimal::from_str` is stricter and will fail. Frontend says OK, server says no. **Fix:** add a stricter amount-pattern regex on the form.
- **Tab-switch during capture race.** User clicks Photo, picks file, then taps a different top-level tab before the ~1.2s extract round trip completes. `FinancesPage` unmounts; the `spawn`ed task may try to `state.set(...)` on a Signal whose scope no longer exists. Dioxus 0.7.2 likely cancels spawned futures on scope drop but I haven't verified the exact semantics. **Fix scope:** verify and add a stop-token if needed; minimum, ensure no panic on dropped-scope writes. **Broader rework deferred to Cycle 4** (see `tasks.md` → "Cross-submodule state management"): lift in-flight captures to a top-level shared store keyed by capture-id so cross-tab navigation doesn't destroy in-flight work.
- **`cargo tauri dev` runtime path NOT tested.** Carries forward from the Phase 2 known-limitations list. Frontend mock branch is the only thing exercised; the real Tauri-IPC branches for `extract_document` and `record_transaction` are unobserved.
- **Real Gemini extraction unverified.** The mock returns a canned `Loblaws — Groceries` receipt regardless of input. Actual Gemini response shape, confidence calibration, and error modes against a real Walmart receipt photo are unknown. The `extract_document` server endpoint compiles + boots, but no live round trip has happened.

### UX gaps worth knowing

- **No "saved" confirmation.** After `Save transaction`, the form unmounts and the user lands on Finances Home with no toast / banner. Transaction "vanishes" until Phase 4 transactions list lands. **Fix:** small ephemeral toast in `HomeView` ("Saved at HH:MM") that fades after 3s.
- **No account / commodity autocomplete.** Users type `Expenses:Groceries` by hand for every transaction. `list_accounts` Tauri command exists from Phase 1 but is not wired into the form. **Fix:** `use_future` to load accounts on mount + datalist-based autocomplete.
- **PDF retry uses stale hint.** If the user picks `bank_statement`, upload fails, then changes the radio to `paystub`, clicking Retry uses the original `bank_statement` (value was captured into the error variant at failure time). Either is defensible; document the choice or read the hint signal fresh inside the retry closure.
- **No network timeout on the `reqwest` client call.** If the server hangs (slow Gemini, network blip), the Tauri command + frontend spinner block indefinitely. **Fix:** `reqwest::ClientBuilder::default().timeout(Duration::from_secs(45))` on `AppState.http` init.
- **Mobile touch-target concerns.** `Remove` button on posting rows is ~24px (below iOS 44px / Android 48px guideline). `capture="environment"` on the photo input is unverified on actual Android.
- **Form date validation laxness.** HTML date input rejects format errors in modern browsers but doesn't catch invalid dates like 2026-02-31 in some configurations. Frontend says OK; backend `NaiveDate` parse fails.

### Architecture / deferred (Cycle 4 polish territory)

- **3.3 share-target — manifest-only.** Edit sits in `gen/android/.../AndroidManifest.xml` which is gitignored. Durable home requires Tauri 2 override convention (likely `tauri.conf.json` `android.manifestExtras` or a side-by-side template). The Rust intent-handler glue is a separate item; needs `tauri-plugin-deep-link` or custom JNI. **Defer to a dedicated Android session** with the APK build pipeline active.
- **Frontend types duplicate backend types.** `ExtractedDraft` / `TransactionFormDraft` / `PostingInput` mirror `core::extraction::ExtractionResult` / `commands::budget::TransactionDraft` / `core::events::Posting`. No shared types crate — drift will eventually bite (renames silently break deserialization). Per [[feedback-breaking-changes-ok]] this is acceptable pre-daily-use.
- **No tests for the Phase 3 code.** Bridge mock functions, `finances.rs` components, `extract_handler` route, `extract_document` Tauri command — all untested. Per [[feedback-test-gap-audit-yield]] the user prefers fixes-with-tests-inline; nothing's broken yet, so deferred.
- **Server `/documents/extract` has no auth.** Same situation as sync routes — acceptable behind Tailscale per [[project-auth-deferred]]. Worth calling out separately because LLM calls cost money: an exposed unauthed server burns Gemini credits, not just CPU.
- **`Vec<u8>` Tauri IPC inefficiency.** Bytes serialize as JSON arrays — ~3× wire overhead for multi-MB photos. Tempfile-based handoff or Tauri's binary IPC would be faster but adds coupling. Acceptable for Phase 3 MVP.
- **`HintRadio` `'static` constraint.** Component takes `&'static str` for value/label, locking the PDF hint set at compile time. Fine for now; would need refactor if hints ever come from user config.
- **15 MB document body limit.** Modern iPhones produce 5-10 MB receipt photos easily. Heuristic limit; bump if 413s show up.
